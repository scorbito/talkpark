# 경기톡 (Match Talk) — 기획서

> 커뮤니티 탭의 두 번째 게시판. 모든 글이 특정 KBO 경기에 묶이는 가벼운 자유게시판입니다.
> 작성일: 2026-05-13 / 상태: 기획 회의 정리본

관련 문서:
- 기존 후기 기능: [feature-overview.md](./feature-overview.md) §2.4 커뮤니티
- 상위 인덱스: [../WORKPLAN.md](../WORKPLAN.md)

## 1. 왜 만드는가

직관 후기는 "갔다 와야지만 쓸 수 있는" 무거운 글입니다. 자주 못 씁니다.
일반 야구 커뮤니티는 글이 많지만 "어제 LG-두산전 얘기"를 찾으려면 검색 노가다가 필요합니다.

우리는 이미 **KBO 경기 데이터를 다 가지고 있습니다**. 글을 특정 경기에 자동으로 묶을 수 있다는 게 다른 야구 커뮤니티와의 결정적 차이입니다.

→ **경기톡**: 사진/인증 없이 가볍게 한두 줄, 모든 글이 경기에 묶이는 게시판.

---

## 2. 후기와의 차이점

| | 직관 후기 (기존) | 경기톡 (신규) |
|---|---|---|
| 작성 조건 | 직관 인증 필수 (attendance 1:1 종속) | 누구나 (로그인만) |
| 사진 | 1~3장 필수 | **0~1장 (선택)** |
| 본문 | 길게 OK | **300자 제한, 한두 줄 권장** |
| 경기 연결 | attendance → game | 글 작성 시 직접 선택 |
| 작성 시점 | 경기 후 (직관 후) | **경기 전/중/후 언제든** |
| 한 경기당 글 수 | 1개 | **무제한** |
| 톤 | 회고/기록 | 잡담/응원/예측/감정 분출 |

핵심 차별점은 **"인증 필요 없음 + 짧은 글 + 시점 자유"**. 사진 1장 허용해도 위 본질은 유지됩니다.

## 3. 작성 스펙

### 3.1. 작성 가능 날짜
- **이번 주 월~일 7일** (월요일 00시 KST 기준 새 주차 시작).
- 단, 경기가 없는 날(우천 취소, 휴식일 등)은 자동으로 선택지 제외.
- 지난 주 글은 **읽기 전용**으로 남음 (목록/필터 노출은 유지).

### 3.2. 작성 흐름
1. 날짜 선택 (이번 주 7일 중) — 기본값 오늘
2. 그 날짜의 경기 목록에서 1경기 선택
3. 본문 작성 (300자 제한)
4. 감정 태그 1개 선택 (필수)
5. 사진 1장 첨부 (선택)
6. 등록

### 3.3. 감정 태그 (이모지 + 라벨)
| 이모지 | 라벨 | 용례 |
|---|---|---|
| 🎉 | 환호 | 홈런, 승리, 좋은 플레이 |
| 📣 | 응원 | 경기 전/중 응원, 화이팅 |
| 😡 | 분노 | 오심, 부진, 패배 |
| 😰 | 불안 | 동점 상황, 마무리 위기 |

→ 4개로 시작. 사용자 반응 보고 추가/조정.

### 3.4. 박제 데이터 (점수 스냅샷)
글 작성 시점에 `games` 테이블에서 현재 점수/상태를 읽어 글 row에 저장합니다. 이닝은 박제하지 않음 (정확도 보장 어려움).

저장 필드:
- `score_home_at_post`, `score_away_at_post` — 작성 시점 스코어
- `status_at_post` — 작성 시점 경기 상태 (scheduled / in_progress / finished)

cron 정책·표시 분기는 §9 참조.

## 4. 표시 스펙

### 4.1. 글 카드 구성
```
┌─────────────────────────────────┐
│ [작성자 닉네임] [응원팀뱃지] [✓직관 인증]   2시간 전 │
│ ─────────────────────────────  │
│ LG 3 : 2 두산  ·  진행 중           │  ← 박제 스냅샷
│                                 │
│ "역시 우리 팀..."                    │  ← 본문
│ [선택 사진 1장]                       │
│                                 │
│ 🎉 환호                            │  ← 감정 태그
│ ♡ 12   💬 4                       │  ← 좋아요 · 댓글
└─────────────────────────────────┘
```

뱃지 정책:
- **응원팀 뱃지**: 프로필의 내 팀 자동 표시. 같은 경기에 양 팀 팬이 섞이는 게 묘미.
- **직관 인증 뱃지**: 그 사용자가 해당 경기에 attendance 레코드가 있으면 표시. 신뢰도 ↑.

### 4.2. 목록/필터
- 기본 정렬: 최신순
- 필터:
  - 날짜 (이번 주 7일)
  - 경기 (선택 날짜의 경기 목록)
  - (옵션) 응원팀 — "이 경기 + LG팬 글만"
  - (옵션) 감정 태그
- 상단에 선택된 경기의 컨텍스트 헤더(스코어/상태/구장) 노출.

### 4.3. 댓글
- 모든 글에 댓글 가능. 요즘은 표준.
- 후기 댓글과 **공용 컴포넌트로 구현** → 한 번 만들어 두 게시판이 같이 씀.
- 댓글 자체는 글에 종속 (경기에 종속 X).
- 경기톡 상세 페이지는 만들지 않고, 카드형/타임라인 목록 안에서 댓글을 펼쳐본다.

### 4.4. 좋아요
- 후기와 동일 패턴.

## 5. 진입 동선

### 5.1. 커뮤니티 탭 구조
```
커뮤니티
├─ [후기] 탭 (기존)
└─ [경기톡] 탭 (신규)
```

### 5.2. 홈 — 이번 주 일정에서 진입
- 홈에 이미 `weekGames` (이번 주 월~일 경기) 데이터가 있음 → 그대로 활용 [app/page.tsx:38-44](app/page.tsx#L38-L44).
- 각 경기 카드에 **글 개수 뱃지** 노출: 예) `💬 12`
- 카드 탭 → 일정 상세 (또는 직접 경기톡 진입, 5.4 참고).

### 5.3. 일정 → 상세
- 일정 상세 화면의 해당 경기 옆에 **"경기톡 보기"** 버튼 추가.
- 버튼 누르면 그 경기 글만 필터된 목록으로 진입.

### 5.4. 진입 UX 결정 필요 사항
- 경기 카드 누름 → **(a) 일정 상세로 이동 후 거기서 경기톡 버튼** vs **(b) 바로 경기톡 직행**?
- → 현재 권장: **(a)** (일정 상세 흐름과 일관성). 단, 글 개수 뱃지를 길게 누르면 (b)로 가는 단축도 고려 가능.

## 6. 데이터 모델 (초안)

### 6.1. 신규 테이블 `match_posts`
```
id              uuid PK
user_id         uuid FK → users
game_id         uuid FK → games
body            text (max 300)
photo_url       text NULL
emotion_tag     enum('cheer','support','anger','anxiety')
score_home_at_post   int NULL
score_away_at_post   int NULL
status_at_post       enum('scheduled','in_progress','finished')
created_at      timestamptz
updated_at      timestamptz
deleted_at      timestamptz NULL
```

### 6.2. 신규 테이블 `match_post_likes`
- 후기 좋아요와 동일 패턴. (`match_post_id`, `user_id`, `created_at`)

### 6.3. 댓글 — 컴포넌트 공유 + 테이블 분리
- **테이블은 분리**: 후기 = `review_comments`, 경기톡 = `match_post_comments` (별도 신설).
- **컴포넌트는 공유**: `<CommentThread />` 같은 범용 컴포넌트 + props로 target 종류 주입.
- 다형 댓글(`comments(target_type, target_id)`)은 RLS 정책이 복잡해져 도입하지 않음.

**작업 순서 (확정)**:
1. 경기톡 작업 시작 전, 기존 후기 댓글 UI를 범용 컴포넌트로 분리/리팩토링.
2. 경기톡 본 작업 진행 시 같은 컴포넌트 재사용.

이렇게 하면 후기 화면도 영향 없고, 경기톡 댓글 UI는 처음부터 깨끗하게 들어감.

### 6.4. 인덱스
- `(game_id, created_at desc)` — 경기별 최신순 조회용 (가장 흔한 쿼리).
- `(user_id, created_at desc)` — 마이/프로필에서 내 글 조회용.

### 6.5. 직관 인증 뱃지 처리
- 글 카드 렌더링 시 `attendances` 테이블과 `(user_id, game_id)`로 조인.
- 미리 비정규화 컬럼(`is_attended`)을 두지 않음 — attendance가 사후에 추가/삭제될 수 있어 정합성 위험.

## 8. 운영·정책 결정사항

- **악성 글 대응**: MVP에는 신고/숨김 없음. Phase 2에 **신고 버튼** 추가 예정. MVP 출시 전까지는 어드민 강제 삭제로만 대응.

## 9. 라이브 스코어 캐싱 정책

분 단위 cron 폴링을 도입하지 않는다. 대신 **글쓰기 진입 시점의 온디맨드 캐싱(lazy refresh)** 방식으로 라이브 스코어를 가져온다.

### 9.1. 박제 데이터 범위
글에 박제하는 건 **현재 점수와 상태**.
- `score_home_at_post`, `score_away_at_post` — 작성 시점 스코어
- `status_at_post` — 경기 전/중/후 구분 (표시 분기에 필요)
- 이닝은 박제하지 않음 (정확도 보장 어려움, "5회말" 같은 정밀 표기 없음).

### 9.2. 캐싱 흐름
```
사용자가 [경기톡 글쓰기] 진입
  ↓
games.last_live_synced_at < now() - TTL ?
  ├─ YES → KBO API 호출 → games 테이블 갱신 + last_live_synced_at = now()
  └─ NO  → DB 캐시 그대로 사용
  ↓
화면에 "현재 점수: LG 3 : 2 두산" 표시
사용자가 작성하는 동안 그 값 유지
  ↓
[등록] → 표시되던 값을 match_posts.score_*_at_post 에 박제
```

### 9.3. 캐시 TTL
- **권장 TTL: 2~3분**.
- 야구는 1분 안에 점수 바뀌는 일이 거의 없으므로 충분한 정확도.
- 너무 짧으면 KBO API 호출 빈도 ↑, 너무 길면 점수 부정확.

### 9.4. 운영상 이점
- cron 분 단위 폴링 불필요 → Vercel 한도/외부 cron 부담 해소.
- 글쓰려는 사용자가 있을 때만 sync → 자원 낭비 zero.
- 인기 경기는 자동으로 자주 갱신, 비인기 경기는 갱신 안 됨 (자연스러운 부하 분산).

### 9.5. 박제 시점 — 진입 vs 등록
- **진입 시점의 값을 박제한다** (= 화면에 보여준 값).
- 사용자가 작성하는 동안 실제 경기 점수가 바뀌더라도 박제값은 진입 시 값 유지.
- 이유: "사용자가 보면서 글을 쓴 점수" 와 "박제값" 이 일치해야 혼란이 없음.

### 9.6. 동시 작성자 / 인플라이트 요청
- MVP는 **단순 구현**: 캐시 miss 시 각자 API 호출. 1~2초 안 동시 호출은 인기 경기일 때만 발생하는 사치스러운 문제.
- 추후 부하 보고 single-flight 패턴 도입 검토.

### 9.7. KBO API 실패 대응
- 호출 실패 → DB 캐시값 그대로 사용 (stale이어도 OK).
- 캐시도 없으면 → 점수 박제 없이 글 작성만 허용 (`score_*_at_post = NULL`).
- **글쓰기를 막지 않는 게 핵심**. 점수는 부가 정보.

### 9.8. 기존 cron과의 관계
- **점수 스냅샷 목적의 cron은 폐기.**
- 단, **일정/결과 sync용 cron은 그대로 유지** — 일정 페이지, 홈, 후기 결과 확인 등에서 필요.
- 메모리에 적힌 "cron-job.org 분 단위 백업"은 이번 결정에 한해 불필요.

### 9.9. 표시 분기
| status_at_post | 글 헤더 표시 예 |
|---|---|
| scheduled | "경기 전" (스코어 미표시) |
| in_progress | "LG 3 : 2 두산 · 진행 중" |
| finished | "최종 LG 5 : 3 두산" |

### 9.10. 데이터 모델 영향
`games` 테이블에 컬럼 1개 추가:
```
last_live_synced_at  timestamptz NULL  -- 라이브 스코어 마지막 갱신 시각
```

## 10. 차후 확장 아이디어 (Backlog)

- **시점별 글 가치 표시**: 경기 전 예측글이 경기 후에도 의미 있게 보이도록 "예측 적중/실패" 표시.
- **명장면 태그**: 글에 "역전 만루홈런" 같은 명장면 태그 + 같은 태그끼리 묶어보기.
- **경기 종료 후 자동 정리 카드**: 한 경기 끝나면 "오늘 이 경기 톡 베스트 3"을 카드로 만들어 공유.
- **선발투수 자동 노출**: KBO API에서 라인업/선발 데이터를 추가로 가져와 글 헤더에 표시.
- **이닝별 글 보기**: 점수 스냅샷 활용해서 "1~3회 분위기", "9회 분위기" 같은 시간 필터.

## 11. 필터 상태의 타임라인 보기

경기톡 기본 피드는 카드형 목록을 유지합니다. 날짜 또는 특정 경기 필터가 적용된 경우에도 기본은 카드형 목록으로 보여주고, 사용자가 원할 때만 `타임라인 보기`로 전환할 수 있게 합니다.

### 11.1. 기본 원칙

- 필터는 글 범위를 좁히는 기능이다.
- 타임라인 보기는 같은 글 목록의 표시 방식을 바꾸는 기능이다.
- 필터 적용 시 자동으로 타임라인으로 바꾸지 않는다.
- 필터 상태 헤더에 `필터 해제` 버튼과 함께 `타임라인 보기` 버튼을 노출한다.
- 타임라인 모드에서는 `카드로 보기` 버튼으로 되돌아갈 수 있다.

예시:

```text
5/13 경기톡
두산 vs KIA 외 4경기
```

### 11.2. 날짜 필터와 경기 필터

날짜 필터:

- 해당 날짜의 경기톡만 보여준다.
- 여러 경기가 있을 수 있으므로 경기별 섹션으로 묶는다.
- 각 경기 안에서 경기 상태별 타임라인을 보여준다.

특정 경기 필터:

- 해당 경기의 경기톡만 보여준다.
- 경기 정보는 상단 헤더에 한 번 표시한다.
- 아래에는 경기 상태별 섹션만 보여준다.

### 11.3. 타임라인 구조

타임라인은 경기 상태/시점을 섹션 헤더로 사용하고, 그 아래에 해당 상태에서 작성된 글들을 시간순으로 쌓습니다.

특정 경기 필터 예시:

```text
두산 vs KIA
5/13 · 잠실 · 최종 2:9

경기 전
────────────────
[프로필] 오늘은 승요!
오늘도 화이팅!

[프로필] 두산러버
오늘 선발 믿습니다.

최종 · 두산 2 : 9 KIA
────────────────
[프로필] 오늘은 승요!
무기력하게 졌네 두산.

[프로필] 호랑이팬
오늘 KIA 타선 진짜 좋았다.
```

날짜 필터 예시:

```text
5/13 경기톡

두산 vs KIA
  경기 전
    오늘은 승요!: 오늘도 화이팅!
  최종
    오늘은 승요!: 무기력하게 졌네 두산.

삼성 vs LG
  최종
    삼성팬: 오늘 타선 미쳤다!
```

### 11.4. 타임라인 글 표시

타임라인의 각 글은 가볍게 훑는 것을 목표로 합니다.

표시 요소:

- 사용자 프로필 사진
- 닉네임
- 작성 시간
- 감정 태그
- 본문
- 좋아요 수
- 댓글 수

사진 처리:

- 카드형 보기에서는 기존처럼 글당 사진 1장을 표시할 수 있다.
- 타임라인 보기에서는 사진을 직접 표시하지 않는다.
- 사진이 있는 글은 `사진 있음` 또는 `📷` 표시만 제공한다.
- 사진은 카드형 보기에서 확인한다.

액션 정책:

- 좋아요는 개별 글 단위로 유지한다.
- 댓글은 개별 글 단위로 유지하며, 타임라인 글 아래에서 펼쳐볼 수 있게 한다.
- 삭제도 개별 글 단위로 유지한다.
- 타임라인에서는 액션 버튼을 작게 표시하되, 별도 상세 페이지로 이동하지 않는다.

### 11.5. 정렬 기준

타임라인은 기본적으로 작성 시점 오름차순으로 보여줍니다.

- 경기 전 글 → 진행 중 글 → 최종 글 순서가 자연스럽게 보이도록 한다.
- 같은 상태 안에서는 작성 시간 오름차순을 우선한다.
- 카드형 기본 피드는 기존처럼 최신순을 유지한다.

### 11.6. 구현 메모

첫 구현은 DB 구조 변경 없이 현재 `match_posts` 목록을 클라이언트 또는 서버 조회 결과에서 그룹핑해 표시하는 방식으로 시작할 수 있습니다.

그룹핑 기준:

```text
날짜 필터: game_id → status_at_post → posts[]
경기 필터: status_at_post → posts[]
```

추후 필요하면 경기톡 전용 타임라인 컴포넌트로 분리합니다.
