# 시즌 레벨 기획

> 직관 등록과 후기 작성을 자연스럽게 유도하기 위한 시즌제 성장 기능 기획입니다.
> 이 문서는 시즌 레벨의 제품 정책을 정리한 기획 문서입니다. 실제 DB 구조, 액션 연결, 백필 절차, 테스트 범위는 별도 개발 문서에서 다룹니다.

- 버전 관리: [../VERSION.md](../VERSION.md)

## 1. 목적

시즌 레벨 기능의 목적은 공정한 경쟁 랭킹이 아니라 앱 활성화입니다.

- 직관 등록을 습관화한다.
- 후기 작성을 유도한다.
- 마이페이지와 커뮤니티에서 성장감을 제공한다.
- 매 시즌 다시 쌓는 재미를 만든다.
- 티켓 인증 사용자에게 더 빠른 성장 보상을 준다.

## 2. 기본 컨셉

「오늘은 승요」는 직관 앱이라는 정체성을 유지합니다.

- 앱 안에서는 `집관`이라는 명칭을 별도로 쓰지 않는다.
- 사용자가 어떤 경기를 자기 직관 기록으로 남길지는 자유롭게 두되, 경기일 기준 하루 1개만 기록할 수 있다.
- 티켓 인증은 필수가 아니라 신뢰도와 성장 보너스 요소로 둔다.
- 시즌 레벨은 “한 시즌 동안 앱에 직관과 후기를 얼마나 꾸준히 기록했는지”를 보여주는 활동 지표다.

사용자 설명 문구 초안:

```text
직관을 기록하고 후기를 남길수록 시즌 레벨이 올라가요.
티켓 인증을 하면 더 빠르게 성장할 수 있어요.
```

## 3. XP 획득 정책

기본 XP:

| 행동 | XP | 메모 |
|---|---:|---|
| 직관 등록 | 30 | 사용자가 자유롭게 기록할 수 있는 기본 활동 |
| 후기 작성 | 70 | 직관당 1개만 작성 가능 |
| 티켓 인증 | 100 | 인증 성공 시 추가 보상 |
| 사진 포함 후기 | 20 | 사용자가 직접 업로드한 사진이 있는 후기만 1회 보너스 지급 |

풀세트 1경기 기준:

```text
직관 등록 30 + 후기 작성 70 + 티켓 인증 100 + 사진 포함 후기 20 = 220 XP
```

Lv.10 기준은 풀세트 약 35경기입니다.

```text
35경기 × 220 XP = 7,700 XP
```

### 3.1. XP 획득 시점

XP는 행동이 저장되거나 인증되는 시점에 지급하되, 직관 등록 XP는 경기 종료 확인과 연결합니다.

| 행동 | 지급 시점 | XP |
|---|---|---:|
| 직관 등록 | 경기 종료 후 결과 확인 시점 | 30 |
| 티켓 인증 | 인증 성공 시점 | 100 |
| 후기 작성 | 후기 저장 성공 시점 | 70 |
| 사진 포함 후기 | 사용자 업로드 사진이 처음 포함된 시점 | 20 |

직관 등록 XP는 등록 즉시 지급하지 않습니다. 미래 경기를 미리 등록하는 경우가 있으므로, 경기 종료 후 사용자가 결과를 확인했을 때 지급합니다.

```text
직관 등록
→ 예정/진행/종료 상태 유지
→ 경기 종료 후 결과 확인
→ 직관 등록 XP +30
```

### 3.2. 티켓 인증 케이스

티켓 인증은 직관 등록 시 함께 하거나, 직관 등록 후 나중에 할 수 있습니다. 두 경우 모두 인증 성공 시점에 `+100 XP`를 1회 지급합니다.

케이스 1. 직관 등록할 때 티켓도 같이 등록:

```text
티켓 사진 첨부 + 직관 등록 저장
→ 저장 성공
→ 티켓 인증/매칭 성공
→ 티켓 인증 XP +100
```

케이스 2. 직관 등록 후 나중에 인증:

```text
직관 등록 저장
→ 나중에 티켓 사진 추가
→ 티켓 인증/매칭 성공
→ 티켓 인증 XP +100
```

이미 경기 결과 확인으로 직관 XP를 받은 뒤 나중에 인증하더라도, 인증이 처음 성공한 경우에는 티켓 인증 XP를 지급합니다.

```text
결과 확인 완료 +30
→ 나중에 티켓 인증 성공
→ 티켓 인증 XP +100
```

티켓 인증 XP 지급 조건:

- 티켓 사진이 첨부되어 있다.
- 직관 기록 저장이 성공했다.
- 티켓 인증/경기 매칭이 성공했다.
- 해당 직관에 `ticket_verified` XP가 아직 지급된 적 없다.

### 3.3. 사진 보너스 케이스

후기는 시스템 기본 이미지를 사용하지 않습니다. 사진이 비어 있는 상태로 시작하고, 모든 사진은 사용자가 직접 업로드한 Storage 객체입니다. 따라서 사진 보너스 판별은 단순히 `reviews.photos.length > 0` 으로 충분합니다.

지급 조건:

- 후기가 저장되어 있다.
- `reviews.photos`에 사진이 1장 이상 있다.
- 해당 직관에 `review_photo_bonus` XP가 아직 지급된 적 없다.

후기 작성 시점에 사진이 있으면 즉시 `+20 XP`를 지급합니다. 후기 작성 후 수정으로 사진을 처음 추가한 경우에도 아직 지급된 적이 없다면 `+20 XP`를 지급합니다.

```text
후기 작성
→ 사진 없음
→ 후기 XP +70
→ 나중에 사진 추가
→ 사진 보너스 XP +20
```

한 번 지급된 사진 보너스는 사용자가 후기에서 사진을 삭제하더라도 회수하지 않습니다. 단, 후기 자체를 삭제하면 후기 작성 XP와 사진 보너스 XP를 함께 회수합니다.

## 4. XP 지급 제한

중복 지급을 막기 위한 최소 정책입니다.

- 한 사용자는 같은 날짜에 직관 기록을 1개만 등록할 수 있다.
  - **DB 레벨**: `attendances` 테이블의 `check_one_attendance_per_day()` trigger로 보장. `games.game_date` 기준이라 우천 연기 동기화 부담 없음.
  - **앱 레벨**: server action에서 미리 체크해 사용자에게 친절한 안내 제공.
- 같은 날짜에 이미 직관 기록이 있으면 새 등록 대신 기존 기록 수정 안내를 우선한다.
- 직관 등록 XP는 해당 날짜의 최초 유효 직관 기록 1회만 지급한다.
- 후기 XP는 직관 기록 1개당 1회만 지급한다.
- 사진 포함 후기 XP는 직관 기록 1개당 1회만 지급한다.
- 티켓 인증 XP는 직관 기록 1개당 1회만 지급한다.
- 티켓 인증은 이미지 hash 중복 차단을 유지한다.

사용자 안내 문구 초안:

```text
하루에 하나의 직관만 기록할 수 있어요.
이미 이 날짜에 등록한 직관이 있어요. 기존 기록을 수정해 주세요.
```

## 5. XP 회수 정책

삭제와 재등록을 고려해 XP는 이벤트 로그 기반으로 관리하는 것이 안전합니다.

정책:

- 직관 기록을 삭제하면 직관 등록 XP를 회수한다.
- 연결된 후기가 삭제되면 후기 작성 XP와 사진 포함 보너스를 회수한다.
- 직관 기록 삭제로 티켓 인증이 사라지면 티켓 인증 XP도 회수한다.
- 후기 수정으로 사진을 삭제하더라도 사진 보너스 XP는 회수하지 않는다.
- 재등록하면 현재 정책에 따라 XP를 다시 지급한다.
- 삭제/재등록 반복이 있어도 최종 XP는 현재 유효한 활동 상태와 일치해야 한다.

구현 방향 초안:

```text
xp_events
- user_id
- season
- type
- source_id
- xp
- created_at
```

예시 이벤트:

```text
attendance_result_acknowledged +30
attendance_result_acknowledged_revoked -30
review_created +70
review_deleted -70
review_photo_bonus +20
review_photo_bonus_revoked -20
ticket_verified +100
ticket_unverified -100
```

현재 시즌 XP는 `xp_events.xp` 합산으로 계산합니다.

### 5.1. XP 감소 케이스

획득 케이스의 반대 행동은 해당 활동으로 지급된 XP를 회수합니다.

| 사용자 행동 | 회수 대상 |
|---|---|
| 티켓 인증 삭제/취소 | 티켓 인증 XP `-100` |
| 후기 삭제 | 후기 작성 XP `-70`, 사진 보너스 XP `-20` |
| 후기 수정으로 사진 삭제 | 회수 없음 |
| 직관 기록 삭제 | 해당 직관에서 발생한 모든 XP |
| 티켓 인증된 직관 삭제 | 직관 XP, 티켓 인증 XP, 연결 후기 XP, 사진 보너스 XP |

중요한 원칙:

- 고정 금액을 무조건 빼지 않는다.
- 해당 source에서 실제 지급된 XP 이벤트가 있는 경우에만 반대 이벤트를 생성한다.
- 이미 회수한 XP는 중복 회수하지 않는다.
- XP 합계는 현재 유효한 직관/인증/후기 상태와 일치해야 한다.

예시 1. 티켓만 삭제:

```text
기존 이벤트:
ticket_verified +100

회수 이벤트:
ticket_unverified -100
```

예시 2. 후기 삭제:

```text
기존 이벤트:
review_created +70
review_photo_bonus +20

회수 이벤트:
review_deleted -70
review_photo_bonus_revoked -20
```

예시 3. 풀세트 직관 삭제:

```text
기존 이벤트:
attendance_result_acknowledged +30
ticket_verified +100
review_created +70
review_photo_bonus +20

회수 이벤트:
attendance_deleted -30
ticket_verified_revoked -100
review_deleted -70
review_photo_bonus_revoked -20
```

예시 4. 아직 결과 확인을 하지 않은 직관 삭제:

```text
기존 이벤트:
ticket_verified +100

회수 이벤트:
ticket_verified_revoked -100
```

이 경우 `attendance_result_acknowledged +30`이 지급된 적 없으므로 `-30`은 생성하지 않습니다.

### 5.2. 재등록 정책

삭제 후 재등록하면 현재 정책에 따라 XP를 다시 지급합니다.

```text
직관 삭제
→ 기존 지급 XP 회수
→ 같은 날짜 또는 다른 날짜로 재등록
→ 조건 충족 시 XP 재지급
```

사용자에게는 삭제 시점에 아래 안내를 보여주는 것을 권장합니다.

```text
이 직관을 삭제하면 관련 경험치도 함께 회수돼요.
```

후기가 연결된 경우:

```text
이 직관을 삭제하면 연결된 후기와 경험치도 함께 정리돼요.
```

## 7. 시즌 정책

- XP와 레벨은 시즌 단위로 계산한다.
- 새 시즌이 시작되면 새 시즌 레벨은 Lv.1부터 시작한다.
- 지난 시즌의 최고 레벨, 칭호, 총 XP는 기록으로 보관한다.
- 시즌 기준은 경기일(`games.game_date`)의 **연도(int)** 기준으로 계산한다 (`extract(year from games.game_date)`).
- KBO 정규시즌은 보통 3월 ~ 11월이지만, 안전하게 **12/31 23:59:59 KST**까지를 같은 시즌으로 본다. 시즌 종료 후 12월에 등록하는 과거 경기도 그 경기의 연도 시즌으로 계산.
- 같은 연도 안의 기록은 정규시즌, 포스트시즌, 기타 경기 구분 없이 같은 시즌 XP로 본다.
- 추후 시즌 타입을 세분화하면 정규시즌/포스트시즌 분리 여부를 다시 검토한다.

지난 시즌 기록 예시:

```text
2026 시즌 Lv.8 승리요정
총 5,320 XP
직관 기록 27개
후기 22개
인증 18개
```

## 8. 시즌 중 도입과 백필 정책

시즌 레벨 기능은 시즌 중간에 도입될 수 있습니다. 이 경우 기존 사용자들이 손해를 보지 않도록 현재 시즌의 기존 직관/티켓 인증/후기 데이터를 기준으로 XP를 소급 반영합니다.

백필 요약:

- 기능 배포 직후 운영자가 현재 시즌 전체 데이터를 대상으로 1회 백필한다.
- 이후 사용자 로그인 또는 앱 진입 시 유저 단위 보정 백필을 수행한다.
- 백필은 멱등하게 동작해야 하며, 같은 XP 이벤트를 중복 생성하지 않는다.
- 삭제된 직관/후기처럼 현재 DB에 남아 있지 않은 기록은 백필하지 않는다.

상세 정책은 [season-level-backfill.md](./season-level-backfill.md)를 기준으로 합니다.

## 9. 노출 위치

원칙: **본인 공간에만 상시 표시. 타인 정보는 사용자가 명시적으로 눌렀을 때만 모달로.**

상시 노출 위치 (본인 거):
- 마이페이지 프로필 영역
- 홈 내 카드
- 마이페이지 내 시즌 기록 영역

타인 레벨 노출 (눌렀을 때만):
- 후기 상세 / 후기 카드 / 댓글 / 경기톡 글·댓글 등에서 닉네임 또는 프로필 사진을 누르면 **프로필 모달**이 열리고 거기서만 레벨을 본다.
- 상세 정책은 §12 참조.

비노출 위치:
- 커뮤니티 후기 리스트 (스캔성 우선)
- 후기 상세 작성자 영역 (상시 표시 제거 — 모달로만)
- 모든 댓글 작성자 영역
- 경기톡 글·댓글 작성자 영역

표시 예시:

```text
Lv.6 응원단골
```

좁은 공간(홈 내 카드 등)에서는 축약형을 사용할 수 있습니다.

```text
Lv.6
```

## 10. 화면별 디자인 방향

### 10.1. 마이페이지

마이페이지는 기존의 직관 승률/승패무 중심 카드보다 시즌 레벨 정보를 더 크게 보여줍니다. 직관 승률과 승패무는 홈에서도 확인할 수 있으므로 마이페이지에서는 보조 정보로 낮춥니다.

마이페이지 핵심 표시 정보:

- 현재 레벨
- 현재 칭호
- 현재 시즌 XP
- 다음 레벨까지 필요한 XP
- 경험치 진행 바
- 지난 시즌 기록 진입점

프로필 영역 조정:

- 현재 연동 계정 정보는 설정 화면으로 이동합니다.
- 프로필 편집 버튼은 프로필 사진/닉네임 라인 안에 작게 배치합니다.
- 프로필 상단은 `프로필 정보 + 시즌 레벨 카드`가 자연스럽게 이어지도록 구성합니다.

마이페이지 예시 정보 구조:

```text
프로필 사진  닉네임  [편집]
Lv.6 응원단골

시즌 레벨
Lv.6 응원단골
2,980 / 3,800 XP
[██████░░░░]
다음 레벨까지 820 XP
```

### 10.2. 홈

홈의 내 카드에는 레벨과 칭호를 짧게 표시합니다. 홈은 여전히 다음 직관, 현재 직관, 직관 승률을 빠르게 확인하는 화면이므로 레벨 정보는 과하게 크게 만들지 않습니다.

표시 예시:

```text
Lv.6 응원단골
```

### 10.3. 커뮤니티 후기 리스트

후기 리스트 카드에는 레벨을 표시하지 않아도 됩니다.

이유:

- 리스트는 스캔성이 중요합니다.
- 닉네임, 팀 배지, 경기 정보, 후기 이미지, 액션 버튼만으로도 정보 밀도가 높습니다.
- 레벨/칭호까지 넣으면 카드가 복잡해질 수 있습니다.

### 10.4. 후기 상세

후기 상세 작성자 영역에는 **레벨을 상시 표시하지 않습니다**. 닉네임/프로필 사진을 누르면 **프로필 모달**이 열리고 거기서 레벨/칭호를 확인합니다. (§12)

이유:
- 상대적으로 레벨이 낮은 사용자가 위축감을 느낄 수 있음.
- 후기 본문에 집중하는 스캔 흐름을 유지.
- 호기심 있는 사람만 모달로 능동 확인하는 게 부담이 적음.

### 10.5. 댓글 · 경기톡

같은 원칙. 댓글/경기톡 글/경기톡 댓글의 작성자 영역에 레벨을 상시 노출하지 않습니다. 닉네임/사진 탭 → 프로필 모달로 통일.

## 12. 프로필 모달과 자기소개

후기 상세, 후기 카드, 댓글, 경기톡 글/댓글에서는 레벨을 상시 노출하지 않고, 작성자 닉네임 또는 프로필 사진을 눌렀을 때 프로필 모달에서 확인합니다.

프로필 모달은 시즌 레벨, 자기소개, 활동 지표, 친구 신청 동선을 제공하는 별도 기능으로 분리합니다.

상세 기획은 [profile-modal.md](./profile-modal.md)를 기준으로 합니다.

- 특정 행동 기반 뱃지

- 프로필 모달에 최근 후기 미리보기
